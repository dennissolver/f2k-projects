# Design: Admin Reports — voice + LLM over a bounded report catalog

Generated by /office-hours on 2026-06-16
Branch: feat/estate-analytics-phase1
Repo: f2k-projects
Status: APPROVED
Mode: Startup (intrapreneurship — internal F2K admin tool, dogfood for the future distributor platform)

## Problem Statement
Admins (Uwe operationally, Dennis as the fallback) regularly need specific cuts of the
registration / communications data that the fixed admin pages don't surface. Today those asks
route to Dennis, who hand-queries or exports manually. The feature: a **Reports** page where an
admin can get the report they need — on screen for review and as a downloadable file (CSV / PDF) —
driven by a Morgan voice agent that elicits the request, with a plain filter form as the fast path.

## Demand Evidence
Real, recurring, named asks from Uwe (verbatim shape):
1. "Can I get a report that shows all registrations for Branscombe within a date range?"
2. "Registrations for an agent within a date range."
3. "Show me all emails sent to registrants AND the matching emails agents get telling them one of
   their clients has registered."
4. "I need a download of Seafields registrations between these dates."
5. "I need a PDF report showing all registrations and a graph showing days registered, for the funders."

These are recurring operator requests, not hypotheticals — strong demand. They also reveal the
catalog (below): every ask is registrations/comms, filtered by estate/date/agent, output as
screen/CSV/PDF.

## Status Quo
Uwe asks Dennis → Dennis manually queries Supabase / exports / eyeballs the admin pages → sends Uwe
the result. Recurring manual toil on the operator's critical path; doesn't scale as estates grow.

## Target User & Narrowest Wedge
**User:** Uwe (estate operator) primary; Dennis secondary. **Wedge:** a 3-block report catalog —
(a) **Registrations** (filters: estate, date range, agent), (b) **Communications/email log**
(registrant confirmations + the agent "your client registered" notifications), (c) **Funder summary**
(registrations + a registrations-over-time chart → branded PDF). Outputs: on-screen table / CSV / PDF.
These three blocks cover all 5 asks.

## Constraints
- Regulated tier: buyer PII; prior anon-exposure P0 (migration 0027). No LLM-authored SQL against
  prod; read-only, admin-gated, structured-params-only.
- Consume the portfolio voice stack (`@caistech/elevenlabs-convai`, Morgan) — no raw CDN embed.
- PDF via `@caistech/report-generator` (branded). Reuse `/admin/registrations` filters + `audit_log`
  email-send events.

## Premises
1. The 5 real asks are all **parameterised** (pick a block, set filters, choose a format) — none
   requires natural language to express. **Agreed** (evidenced by the asks themselves).
2. Therefore the voice/LLM layer is **experience the asks didn't demand** for v1 — a thin-MVP
   over-build risk. **User overrode this** (chose to build voice + LLM now anyway — see Decision).
3. Doing nothing = continued manual toil on Dennis. Real, recurring pain. **Agreed.**
4. A bounded building-block catalog (not free SQL) is the only safe way to let an LLM near
   PII-laden prod data. **Agreed — non-negotiable.**

## Approaches Considered
### Approach A: Parameterised report builder (minimal viable)
Reports page, 3-block catalog, filter form (estate/date/agent), output table + CSV + PDF. No voice,
no LLM. Covers 100% of Uwe's asks. Effort S/M, Risk Low.

### Approach B: Voice + LLM over the catalog (the original vision) — CHOSEN
Morgan voice elicits the request → LLM maps it to catalog building blocks + structured params →
render + export. Plus a SayFix-style loop: a request the catalog can't serve is captured → drafts a
PR adding a new block → reviewed/merged → becomes a catalog block. Effort L, Risk Med.

### Approach C: Hybrid (A now, voice/LLM seam for later)
Ship A's engine now; structure the catalog so B bolts on when there's evidence anyone wants to talk
to it. Effort M, Risk Low. (Was the office-hours recommendation; not chosen.)

## Recommended Approach
**Chosen by the operator: Approach B — voice + LLM now.** Build it in a sane sequence so it is not
fragile:
1. **The engine first** — the 3-block catalog as vetted, parameterised, read-only, RLS-respecting
   queries (the building blocks), with on-screen render + CSV + PDF (report-generator) export. This
   IS Approach A's engine and covers every current ask on its own.
2. **Two front-ends over the one engine** — a plain **filter form** (the fast path: three dropdowns
   + export, for Uwe on a busy day) AND the **Morgan voice agent** (elicits intent → LLM emits the
   same structured params). Never let voice be the only door.
3. **The SayFix novel-report loop — PHASE 3, explicitly OUT of the first ship.** Premise 2 records
   that the catalog covers 100% of current asks, so this loop serves zero current demand; it is
   dogfood for the future platform and ships only after the engine + both front-ends are validated.
   When built: an unsupported ask is captured → drafts a PR adding a new building block → **a human
   (Dennis) rewrites/vets it into a parameterised query on the read-only views and merges it.** The
   LLM proposes capability; the merged code is human-authored. This human-review gate is exactly what
   satisfies the "zero LLM-authored SQL reaches prod" invariant below — an LLM-drafted block is not
   prod until Dennis has vetted it.

Security spine (all phases): structured params validated by Zod, **never LLM-authored SQL in prod**
(the Phase-3 PR loop's human review is the gate that guarantees this); read-only; admin-gated
(`middleware` + `getAdminUser` + `admin_users`); build blocks on read-only reporting views that
pre-aggregate / strip PII. Concrete limits (statement timeout, row cap, the exact PII columns to
strip per view) are named TODOs for the eng review — suggested starting defaults: 10s statement
timeout, 50k-row cap, views exposing no raw email/phone unless the report's purpose requires it.

## Data sources & build order
Per block (confirm exact tables/views in the eng review):

  `branscombe_registrations`, `wavecrest_registrations`, `dutton_registrations`); a **cross-estate
  read-only union view is a prerequisite** for the estate filter (the tables are per-estate, not

- **(b) Communications/email log** — covers the emails that actually exist: the **registrant
  confirmation** + the **admin notification** (both fire from the register routes). Source TBD:
  reconstruct from `audit_log` email-send actions, or add a dedicated email-events log. **The
  registrant+admin half is gated on that source question.** The **agent-notification half does NOT
  exist yet** — see the finding below; once built it joins this log automatically.
- **(c) Funder summary** — registrations (block a's view) + a registrations-over-time series; PDF via
  `@caistech/report-generator`.

**Build order:** blocks **(a) and (c) ship independently** of the email-log open question; only block
**(b)** is gated on it. The first ship = engine + filter form + voice over blocks (a) and (c); block
(b) follows once its data source is confirmed; the SayFix loop is Phase 3.

## Finding (2026-06-16): the agent-notification email does not exist
Ask 3 assumed agents get an email when their referred client registers. **They don't.** The register
routes send only two emails (registrant confirmation + admin notification — verified in
`src/app/api/seafields/register/route.ts`). The agent profile
(`src/app/agent/(portal)/profile/page.tsx:41-42`) already exposes `notify_new_client` /
`notify_status_change` toggles, but **nothing ever sends them — the toggles are a silent no-op** (a
latent agent-portal bug: an agent who turns them on gets nothing).

**Decision: build the agent-notification email** (its own agent-portal task, NOT part of the Reports
build) — on registration with a `referrer_agent_id`, if `agent.notify_new_client`, send "your client
X registered for lot Y"; mirror for status changes against `notify_status_change`. This honours a
toggle agents already see, makes the portal genuinely useful, and makes ask 3 fully answerable. Once
shipped, agent emails flow into report block (b) automatically.

**New task (agent portal, prerequisite for ask 3's agent half):**
- [ ] **AP1 (P2, human: ~half day / CC: ~30min)** — agent portal — send the referring agent an email
  on client registration (gated on `notify_new_client`) + on status change (gated on
  `notify_status_change`); reuse the templated-email path (`sendTemplated`) + the compliance footer;
  log the send so block (b) can report it. Files: the register routes + status-change path +
  `src/lib/email/`.

# Addendum (2026-06-17): the Funder Demand-Coverage Report — block (c) made lender-grade

Uwe brought a verbatim **lender/credit-team** request that is a much richer version of block (c)
("a PDF report … for the funders"). It is not "registrations + a chart" — it is a
**demand-to-settlement pipeline** a bank would underwrite a construction/retail-mortgage facility
against. It does **not** change the engine-first architecture (Zod params · read-only · PII-stripped
· one engine, form + voice front-ends). It defines the **richest report the engine must serve**, and
makes **gap-handling a first-class feature**, not an afterthought.

## The request, decomposed
> *"whether the demand … is real enough to lend against — not a registration count, but how many
> turn into buyers who can actually settle."* Walk the pipeline by stage with drop-off/conversion;
> map demand against supply **by lot and price tier** (not one headline multiple); de-identified
> borrowing-capacity / income / OO-vs-investor / FHB-scheme bands vs the asking prices; week-by-week
> trend, run-rate, registered→qualified velocity, projected date to the cover trigger; and the
> provenance (verification, dedup, timestamps, every term defined, **which stage "3× cover" counts
> at**).

## The honest data position (the design's spine)
The registration tables capture identity, `lots_selected[]`, `buyer_type`, `finance_status`,
`purchase_timeline`, `created_at`, referral attribution — **all self-declared on the ROI form, with
no verification layer and no post-registration stage tracking.** So:

| Stage Uwe wants | Data status | Engine behaviour |
|---|---|---|
| Registered | ✅ real (`created_at`) | count + timestamps |
| Verified / spoken to | ❌ gap | "Not captured — no contact/verify field" |
| Pre-qualified (borrowing capacity) | ❌ gap | "Not captured — no income/capacity field" |
| Finance pre-approval | ⚠️ **self-declared** (`finance_status` ∈ Pre-approved / Exploring / Cash) | report the self-pick, **labelled unverified** |
| Deposit down | ❌ gap | closest proxy = `intent_locked_to_registration_id` (an admin *intent lock*, not money) — report as proxy, never as "deposit paid" |
| Signed / Settled | ❌ gap | "Not captured" |

**The cover-stage rule (Uwe's explicit "tell me which stage you count 3× at"):** the only stage with
real data is **registered interest**. Therefore **every cover figure is stamped with its counting
stage**, and the headline cover is labelled *"× registered interest (expressions of interest), not
× finance-ready buyers."* That is the single most important honesty in the report; a credit team
that sees an unqualified multiple sold as "buyers" stops trusting the whole paper.

## What the engine answers today (real data)
- **Registered funnel** — count, de-duplicated by person (group by lowercased email; the model
  intentionally allows multi-lot rows per person, so the report de-dups for the funnel and keeps the
  row count for interest-intensity).
- **Coverage by lot & price tier** — the strongest section, fully real: `lots_selected[]` demand
  counted against `seafields_lot_allocations` / `branscombe_unit_allocations` (`retail_price`,
  `stage`, `allocated_to`). Per-lot and per-tier cover ratio; **thin-spots flagged** (supply with
  demand below 1× and demand concentrated on the cheapest tier). *(Only estates with a priced lot
  table — Seafields, Branscombe — support this; others return a gap.)*
- **Buyer mix (self-declared, labelled)** — OO vs investor (`buyer_type`), FHB **count**
  (`buyer_type = First Home Buyer`; scheme *eligibility* is a gap), finance-ready count
  (`finance_status ∈ Pre-approved / Cash`).
- **Trend** — registrations per ISO week, run-rate (trailing-4-week mean), projected date to a
  **registered-interest** cover threshold. Qualified-stage velocity/projection are gaps.

## The gap-surfacing contract (first-class)
Every metric the engine can't satisfy returns a structured **gap** `{ reason, toInstrument }`
instead of a number or a blank (degrade-don't-fake). All gaps roll up into a **"Data to instrument"
appendix** — which **is the lender's collection roadmap** (what to start capturing to make each
missing stage reportable). The report ships day one as ~40% real answers + ~60% structured gaps; each
gap closed upgrades the same report. This is exactly the operator's instruction: *"it's fine if a
query can't be answered — that signifies the gap; surface gaps as we query, don't let them block the
build."*

## Build shape (this addendum)
- **Engine core** `src/lib/reports/funder-demand.ts` — pure aggregations over plain row arrays
  (`buildFunnel`, `buildCoverage`, `buildBuyerMix`, `buildTrend`, `priceBand`, `dedupeByPerson`) +
  the `Metric`/`Gap` types and the cover-stage label. Unit-tested (no DB), mirroring the analytics
  adapter's pure-helper pattern.
- **Data seam** `buildFunderDemandReport(estateSlug, opts)` — fetches registration + lot rows via
  `createSupabaseService()` (read-only) using an estate→table map, feeds the pure builders, assembles
  the report + gap appendix.
- **Surface** `/admin/reports` server component (admin-layout gated) — estate selector + the four
  sections + every gap rendered as a callout + the cover-stage disclaimer + the appendix. CSV/PDF,
  the Morgan voice front-end, and the SayFix novel-report loop stay later phases (engine-first holds).

## Terms (for the credit team — defined in-report)
- **Registration / expression of interest:** one submitted ROI form row. Self-declared, unverified.
- **De-duplicated registrant:** distinct lowercased email across an estate's rows.
- **Cover (at stage S):** demand counted at stage S ÷ available supply. Default S = registered
  interest (the only instrumented stage). Stamped on every figure.
- **Available supply:** lot rows with no firm `allocated_to` buyer (pool labels like GROH/WACHS are
  reservation pools, not sold — see `seafields-allocation-pool-model`).
- **Finance-ready (self-declared):** `finance_status ∈ {Pre-approved by lender, Cash buyer}`. Not
  lender-verified.
- **Intent lock:** an admin-set soft hold (`intent_locked_to_registration_id`) — a commitment
  *proxy*, explicitly **not** a paid deposit.

# Addendum (2026-06-17b): Morgan as the discovery consultant (the interface)

The voice/LLM layer is **not a request translator** — it is a **discovery consultant** that interviews
the admin to a *precise, unambiguous query spec before anything runs*, captures the **why** (not just
the what), and **reframes** the request when the data affords a better answer. This is the whole point
of the LLM interface (operator, 2026-06-17): never pre-guess-and-hand-build a report after the ask,
and never be blind to gaps at request time. The consultant delivers both.

## The discovery loop (mandatory before any query executes)
1. **Resolve every slot to an exact value — no vagueness reaches the engine.** Required slots:
   - **estate** (one of the reportable estates), **window** (explicit dates, not "recent"),
   - **stage** (which pipeline stage the figure counts at — *especially the cover stage*),
   - **grain** (per-lot / per-tier / per-buyer-segment), **dedup** (by person vs raw rows),
   - **cut/segment** (e.g. OO-vs-investor, FHB, finance-ready), **format** (screen / CSV / PDF).
   She does not proceed on an unresolved slot — she asks.
2. **Capture the WHY.** Elicit the underlying intent ("what's this for?"). The why licenses a
   **reframe**: e.g. ask = "registration count," why = "size settlement risk on the mortgage book" →
   she proposes *cover-by-price-tier at the registered-interest stage*, because a raw count doesn't
   answer the real question. The reframe is offered, not imposed — the operator confirms.
3. **Surface gaps live, mid-conversation.** She reads `FUNDER_REPORT_CAPABILITIES` (the manifest in
   `funder-demand-report.ts`) and knows in-the-moment what is real / self-declared / proxy / gap. If a
   slot needs un-instrumented data ("borrowing-capacity bands"), she says so *during discovery*,
   offers the closest proxy (self-declared finance status), and logs the gap to the "data to
   instrument" roadmap — rather than accepting the request and failing later.
4. **Confirm the spec back, then run.** She restates the fully-resolved spec ("Seafields · last 8
   weeks · cover by price tier · registered-interest stage · deduped · PDF — correct?"). Only the
   **operator-confirmed, Zod-validated structured spec** reaches the engine. This confirm-gate **is**
   the security spine: a human-confirmed structured spec is exactly what guarantees *zero
   LLM-authored SQL reaches prod* — the LLM composes primitives + params, never SQL.

## Why this shape
- **Primitives are her vocabulary.** She composes the confirmed spec from the engine primitives
  (funnel / coverage / tiering / buyer-mix / trend) + params. New ask → assembly, not new code.
- **The manifest is her map.** Without it she'd promise vapor and fail at query time; with it she
  steers within the real data and turns every gap into a roadmap line.
- **It mirrors the methodology North Star:** the consultant does the precise workup (discovery →
  unambiguous spec → flag the unknowns), the operator opines and says "go." Morgan is that room,
  pointed at a report request.

## Build note
The consultant is the **next phase** over the shipped engine. It needs: (a) a discovery prompt that
enforces the slot-resolution + WHY-capture + confirm-back loop; (b) the capability manifest passed in
as context so gaps surface live; (c) a Zod `ReportQuerySpec` the confirmed conversation emits, which
the existing seam executes. Reuse Morgan with a per-surface override (as developer/funder/employer do)
— likely no new ElevenLabs agent. The proof surface (`/admin/reports`) is the engine validation; the
consultant adds the door.
